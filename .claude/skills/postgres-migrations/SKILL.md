# PostgreSQL Migrations

## Формат файлов

Каждая миграция состоит из двух файлов:

```
migrations/
├── 20250524120000_create_users.up.sql
├── 20250524120000_create_users.down.sql
├── 20250524130000_create_orders.up.sql
├── 20250524130000_create_orders.down.sql
├── 20250525090000_add_users_phone.up.sql
└── 20250525090000_add_users_phone.down.sql
```

- `*.up.sql` — применение миграции
- `*.down.sql` — откат миграции
- Префикс: UTC-timestamp в формате `YYYYMMDDHHmmss`
- Имя: `<timestamp>_<краткое_описание>` в snake_case
- Создание файлов миграций не требует подключения к БД

## Современный PostgreSQL синтаксис

### Identity вместо SERIAL/BIGSERIAL

```sql
-- Устарело
id BIGSERIAL PRIMARY KEY

-- Правильно (SQL стандарт, PostgreSQL 10+)
id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

`GENERATED ALWAYS AS IDENTITY` — стандарт SQL, в отличие от `SERIAL` который является PostgreSQL-специфичным хаком поверх sequence.

Если нужно иногда вставлять конкретные значения id (миграция данных, импорт):

```sql
id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

### text с ограничением длины

Использовать `text` вместо `varchar(n)`. Для ограничения длины — `CHECK`:

```sql
-- Устарело
name varchar(255)

-- Правильно
name text NOT NULL CHECK (length(name) <= 255)
```

**Правило:** для текстовых колонок всегда ограничивать длину через `CHECK`, кроме случаев
когда длина заведомо неограниченна (описания, комментарии, body).
Если максимальная длина неочевидна — спросить пользователя.

Типичные ограничения:

| Тип данных | Ограничение |
|---|---|
| email | `CHECK (length(email) <= 320)` |
| имя, фамилия | `CHECK (length(name) <= 255)` |
| телефон | `CHECK (length(phone) <= 50)` |
| slug, код | `CHECK (length(code) <= 100)` |
| URL | `CHECK (length(url) <= 2048)` |
| описание, bio | без ограничения или `CHECK (length(bio) <= 10000)` |

### timestamptz вместо timestamp

```sql
-- Неправильно — теряет информацию о часовом поясе
created_at timestamp NOT NULL DEFAULT now()

-- Правильно
created_at timestamptz NOT NULL DEFAULT now()
```

### boolean вместо integer-флагов

```sql
-- Устарело
is_active integer NOT NULL DEFAULT 1

-- Правильно
is_active boolean NOT NULL DEFAULT true
```

### jsonb вместо json

```sql
-- Медленнее, нет индексов
metadata json

-- Правильно — бинарный формат, поддерживает индексы и операторы
metadata jsonb
```

### uuid

```sql
-- Функция gen_random_uuid() встроена с PostgreSQL 13, расширение не нужно
id uuid PRIMARY KEY DEFAULT gen_random_uuid()
```

### enum через CREATE TYPE

```sql
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'cancelled');

CREATE TABLE orders (
    id       bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    status   order_status NOT NULL DEFAULT 'pending'
);
```

## Шаблон up-миграции

```sql
CREATE TABLE users (
    id         bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name       text        NOT NULL CHECK (length(name) <= 255),
    email      text        NOT NULL CHECK (length(email) <= 320),
    phone      text        CHECK (length(phone) <= 50),
    is_active  boolean     NOT NULL DEFAULT true,
    metadata   jsonb,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX ix_users_email ON users (email);
```

## Шаблон down-миграции

down-миграция должна полностью откатывать up:

```sql
DROP TABLE IF EXISTS users;
```

## Правила написания down-миграций

| up-операция | down-операция |
|---|---|
| `CREATE TABLE t` | `DROP TABLE IF EXISTS t` |
| `ALTER TABLE t ADD COLUMN c` | `ALTER TABLE t DROP COLUMN IF EXISTS c` |
| `CREATE INDEX ix` | `DROP INDEX IF EXISTS ix` |
| `CREATE TYPE t AS ENUM (...)` | `DROP TYPE IF EXISTS t` |
| `ALTER TABLE t ALTER COLUMN c SET NOT NULL` | `ALTER TABLE t ALTER COLUMN c DROP NOT NULL` |
| `ALTER TABLE t ADD CONSTRAINT c` | `ALTER TABLE t DROP CONSTRAINT IF EXISTS c` |

Использовать `IF EXISTS` в down-миграциях для идемпотентности.

## Примеры типовых миграций

### Добавление колонки

up:
```sql
ALTER TABLE users ADD COLUMN phone text CHECK (length(phone) <= 50);
```

down:
```sql
ALTER TABLE users DROP COLUMN IF EXISTS phone;
```

### Добавление NOT NULL колонки с дефолтом

up:
```sql
ALTER TABLE users ADD COLUMN role text NOT NULL DEFAULT 'user' CHECK (length(role) <= 50);
```

down:
```sql
ALTER TABLE users DROP COLUMN IF EXISTS role;
```

### Добавление внешнего ключа

up:
```sql
CREATE TABLE orders (
    id      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL REFERENCES users (id),
    total   numeric NOT NULL
);

CREATE INDEX ix_orders_user_id ON orders (user_id);
```

down:
```sql
DROP TABLE IF EXISTS orders;
```

### Добавление enum

up:
```sql
CREATE TYPE payment_method AS ENUM ('card', 'cash', 'crypto');

ALTER TABLE orders ADD COLUMN payment payment_method NOT NULL DEFAULT 'card';
```

down:
```sql
ALTER TABLE orders DROP COLUMN IF EXISTS payment;

DROP TYPE IF EXISTS payment_method;
```

### Добавление значения в существующий enum

up:
```sql
ALTER TYPE payment_method ADD VALUE 'bank_transfer';
```

down:
```sql
-- ALTER TYPE ... ADD VALUE нельзя откатить в транзакции
-- Откат требует пересоздания типа — оставить пустым если значение безвредно
```

## Кастомный мигратор

Мигратор живёт в `cmd/migration/main.go`. Команда `create` работает без подключения к БД —
только создаёт пару файлов `*.up.sql` / `*.down.sql` с timestamp-префиксом.

Использование:
```bash
go run ./cmd/migration create <name>   # создать файлы миграции (без БД)
go run ./cmd/migration up              # применить все новые миграции
go run ./cmd/migration down            # откатить последнюю миграцию
```

Ключевые моменты реализации:
- `Create` — генерирует timestamp + name, создаёт два файла, не трогает БД
- `Up` — читает файлы, сортирует по имени (timestamp обеспечивает порядок), применяет в транзакции через `txManager.Do`
- `Down` — откатывает только последнюю применённую миграцию
- Таблица миграций создаётся автоматически при первом запуске (`EnsureMigrationsTable`)
- Каждая миграция выполняется в отдельной транзакции — при ошибке откатывается только она
- Использовать `filepath.Join` (не `path.Join`) для путей к файлам
